## Plan: Supabase Backend for YSI Ecommerce

This project is a premium fashion e-commerce site with a storefront, checkout flow, customer account area, and an admin dashboard. The current backend layer is mostly simulated with `localStorage` and a mock Supabase client, so the implementation should replace those placeholders with real Supabase Auth, Postgres tables, storage, RLS, server-side data access, and explicit security controls for sanitization, authentication, authorization, and abuse prevention while preserving the existing frontend.

**Steps**
1. Define the Supabase data model and access boundaries. Map the app’s real entities to Postgres tables: products, product variants or attributes, customers, orders, order items, order timeline, inventory logs, admin users, store settings, and any profile tables needed for customer ownership. Decide which data is public, authenticated customer-only, or admin-only, and make authorization rules explicit at the data model level rather than only in the UI.
2. Turn the draft SQL into a production schema migration. Consolidate `supabase-schema.sql`, `seed-admin.sql`, and `scripts/rls-fixes.sql` into an ordered migration flow that creates tables, foreign keys, indexes, timestamps, and triggers. Add explicit grants for `anon`, `authenticated`, and `service_role`, no unsafe public defaults, and a clean seed strategy for the fashion catalog and initial admin account.
3. Harden the database with security-first constraints and policies. Add Row Level Security to every exposed table and view, use ownership or admin membership checks in policies, and avoid deprecated patterns such as `auth.role()`. Ensure updates also have the required select visibility, and avoid `SECURITY DEFINER` except where absolutely necessary and tightly scoped. If helper functions are needed, keep them `SECURITY INVOKER` where possible and place privileged helpers in non-exposed schemas.
4. Replace the mock Supabase client with real SSR-aware clients. Rebuild `src/lib/supabase/client.ts` into proper browser and server client utilities using `@supabase/ssr`, and stop relying on fake JWT generation or local session storage. Add the environment variable wiring expected by the Next.js SSR pattern, and ensure auth state comes from Supabase rather than in-memory or browser-only stubs.
5. Implement authentication and session handling end to end. Update customer login, registration, sign-out, and session hydration to use real Supabase Auth. Use secure session verification on the server, not just cookie presence or raw session objects, and make sure the account area and any protected routes depend on verified identity. If profiles are needed, create or sync them from `auth.users` with sanitized metadata and server-side writes.
6. Move the storefront data reads onto Supabase. Update the shop, product detail, cart/checkout, and account screens so product lists, product detail pages, and account order history read from Supabase instead of `localStorage`. Keep the current UI but change the data source and loading states so the pages still feel immediate and polished.
7. Implement order creation as a backend mutation with strict validation. Replace the checkout page’s local order write with a server-side create path that validates and sanitizes input, inserts the customer record if needed, creates the order row, order items, and initial timeline entry, then returns the order reference for the confirmation page. Preserve the existing payment-method UX if gateway integration is deferred, but persist payment state safely in the database and never trust client-calculated totals without server-side recomputation.
8. Wire admin authentication to real Supabase Auth and admin role checks. Replace the current admin auth provider’s local config checks with a real `auth.users` lookup plus `admin_users` membership verification. Ensure admin pages only render for verified admins, and base authorization on database-backed roles rather than editable client state or user metadata.
9. Migrate admin dashboard reads and writes to Supabase. Replace the dashboard, products, customers, orders, inventory, and settings pages’ `localStorage` lookups with Supabase queries or server actions. Keep the existing dashboard layout and charts, but source the data from real tables and backend queries so updates flow through the database, with every mutation passing through sanitization and authorization checks.
10. Add storage for product imagery and admin upload flow. Create a `product-images` bucket, secure it with bucket-level policies, and update product create/edit paths to store images in Supabase Storage. Decide whether images are public URLs or signed URLs and keep the policy model consistent with how the shop renders product cards and galleries. Include the required INSERT, SELECT, and UPDATE permissions if upsert is needed.
11. Add backend helpers for business logic that should not live in the UI. Centralize operations like slug generation, stock adjustments, order number creation, admin checks, and order timeline updates in server actions, SQL functions, or lightweight API routes. Prefer database constraints and functions where they reduce duplication and protect invariants, and validate all inputs at the boundary before they reach SQL.
12. Add abuse-prevention and operational safeguards. Consider rate limiting or request guards for sensitive mutations, add audit-friendly logging for admin changes, keep service keys server-only, and make sure any privileged route or function is protected by the smallest possible trust boundary. If the app later adds payment webhooks or file upload endpoints, protect them with signature validation and strict input checks.
13. Validate the full data path end to end. Verify public product reads, authenticated customer sign-in, checkout order creation, admin-only dashboard access, storage uploads, and RLS behavior. Add or update a small set of focused tests or smoke checks that confirm the frontend is talking to Supabase rather than the old mock layer.

6. Check RLS with at least one direct query or policy test for each sensitive table before treating the migration as complete.

**Decisions**
- Keep the existing frontend and visual system; the work is a backend/data-layer replacement, not a redesign.
- Treat the current `localStorage` seed data as a migration aid only, not the source of truth.
- Prefer server-side mutations for checkout, admin changes, and storage operations so secrets and elevated privileges never move into the browser.
- Keep customer checkout flexible for now: guest checkout can exist, but order ownership should still be captured by email and/or a linked auth user when available.
- Do not expose admin-only tables through permissive grants; use explicit grants plus RLS.
- Do not trust client-side sanitization or session state alone; treat the server and database as the source of truth for validation and authorization.

**Further Considerations**
1. Decide whether payments will remain manual for the first release or whether you want Paystack / Flutterwave / Stripe server-side checkout next. Manual payment keeps scope smaller; gateway integration adds callbacks, webhooks, and payment status sync.
2. Decide whether product images should be public bucket URLs or signed URLs. Public is simpler for a fashion catalog; signed URLs are stricter but require more server-side handling.
3. Decide whether to model customers as a profile table linked to `auth.users` or keep guest orders separate. Linking is better for account history and future personalization, while guest orders need a fallback email-based lookup.
